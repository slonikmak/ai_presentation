СТУПЕНЬ 1 / ПРОТОТИПЫ
Параллельные версии

Одна спека. Один час.
Три направления.

Запусти 3–5 версий параллельно: из пяти идей три выкидываешь сам, две показываешь коллегам, одна идёт в прод. Это нормально — прототип почти бесплатный.

Raptor 1 → 2 → 3: первая версия страшная, но end-to-end и быстро проверяет гипотезу; вторая лучше; третья — production. Версия 1 — не стыдная.

Критичное правило: прототипы — в отдельных репозиториях, не в боевом. Иначе агент решит «тут так принято» — и будущая генерация скопирует кривые паттерны. Черновики заражают кодовую базу.

Чат — это «детский футбол»

«Все носятся за мячиком, орут — интересно, но результат не всегда такой, как хотелось». Параллельные эксперименты в репах дают сравнимые варианты вместо одной случайной траектории.

СТУПЕНЬ 2 / ПОЧЕМУ СПЕКИ
Потолок чата

на сложности

Миллион токенов контекста лишь отодвигает потолок. На каждый новый блок объясняешь заново, CLAUDE.md превращается в нечитаемый BLOB, plan mode работает до порога.

→
спека в репозиторииmarkdown-файл: версионируется, ревьюится, переживает сессию, переносит знание между агентами
Нужен формат, который переносит знания между сессиями. Это и есть SDD: спека — не церемония, а память проекта.
СТУПЕНЬ 2 / OPENSPEC · данные из доклада
OpenSpec

propose → review →
apply → archive

openspec — workflow
/opsx:propose → proposal · design · tasks
REVIEW ← человек, до первой строчки кода
/opsx:apply → агент идёт по таскам
/opsx:archive → дельта → живая карта спек

Что рождается до кода

proposal — зачем и что · design — как технически · tasks — чеклист. Дёшево поменять здесь — дорого потом. Сильных инженеров — на вход.

8 из 10 — в цель

Два предложения на вход → proposal «вот как это будет в твоей системе». 8/10 попадают, 2 — «ты меня неправильно понял». Лишает магии one-prompt, но радикально режет валидацию.

СТУПЕНЬ 2 / МЕРА · данные из доклада
Масштаб одной спеки

Одна спека ≈ 5 000 строк ≈ 1–2 месяца прежней работы

Цена ошибки в спеке высокая

Спека упаковывает целый квартал. Поэтому ревью спеки — работа сильного инженера, не формальность.

Не пережимать

Детализируешь всё — начинаешь «программировать на английском языке». Спека держит контракты и инварианты, не построчный план.

Читается быстро

Spec-файлы читаются редко; proposal + design — «три странички за пять минут». Это быстрее, чем 30 файлов сгенерированного кода без структуры.

Знакомо по основному докладу: наши принципы спеки те же — лёгкая, только контракты, явный scope, без жёстких рамок имплементации.
СТУПЕНЬ 2 / В ОБЕ СТОРОНЫ
Спеки работают в обе стороны

Не только текст → код, но и код → текст

Прототип → спека

«Изучи код прототипа, собери PRD и спеку. Не пересказывай построчно — зафиксируй INTENT». Фича пересобирается в основном проекте чисто.

Легаси → спеки

«Сканируй проект, сгенерируй спеки уже реализованного; дальше развиваем систему ОТ спек». Старый проект заезжает в SDD без переписывания. LLM отлично читает и обобщает код.

Explore — дистилляция

«Не знаю, чего хочу»: час чата с ресёрчем → сжимается в один proposal → свежий старт без багажа неудачных идей. «Забывать — очень хороший скил».

СТУПЕНЬ 2 / ГЕЙТЫ · кейс из доклада
Гейт №1 · модель данных

Тесты прошли.
Прод рванул на девятой

Спека: «храним токены для разных платформ». Агент «креативно» сделал колонку на платформу. Тесты зелёные на обоих вариантах — рвануло на проде на девятой платформе.

Фикс процесса, а не выговор: в воркфлоу добавлен шаг — модель данных проверяет человек, до кода. Модель данных — один из самых сильных элементов успеха.

credentials.sql
✗ что сделал агент
CREATE TABLE credentials (
  user_id INT,
  twitter_token TEXT, facebook_token TEXT,
  instagram_token TEXT, ... -- +колонка на платформу
);
✓ что должно быть
CREATE TABLE credentials (
  user_id INT, platform TEXT, token TEXT
); -- новая платформа = новая строка
СТУПЕНЬ 2 / ГЕЙТЫ · кейс из доклада
Гейт №2 · reuse / inventory

Восемь реализаций
одного и того же

«Чиним один баг — починилось. Смотришь — а тут не починилось. На третьей итерации: почему мы это чиним три раза? — Смотри, у тебя восемь реализаций». Пять обновили, три забыли. Тесты прошли на всех восьми.

LLM оптимизирует на скорость и срезает углы: написать заново проще, чем найти существующее. → Гейт: полный скан базы «есть готовое?» перед каждой новой фичей.

openspec — gates growing
proposal → design → DATA MODEL ← человек
→ REUSE ← есть готовое? → tasks
гейты появляются там, где налажали
Это наш флайвил правил, встроенный в pipeline: один раз обжёгся → шаг в воркфлоу → больше не повторяется. Воркфлоу растёт из реальных факапов, не из теории.

СТУПЕНЬ 3 / ТИРЫ
Матрица критичности

Не все системы достойны одинакового внимания

Tier 1 · нельзя ломать

Платежи, авторизация, core data.

валидация

Полные спеки · полные тесты · ревью человеком · часы проверки.

Tier 2 · нежелательно

API, интеграции, бизнес-логика.

валидация

Спеки · тесты · точечные проверки.

Tier 3 · потерпит

Админки, внутренние тулзы, дашборды.

валидация

Лёгкие спеки · vibe-code ok · сломается — починим.

Строгость = функция критичности, не догма. Внедрять SDD — с tier 3, потом 2, потом 1. Главный трейд-офф инженера сейчас: где не тратить своё время. Вопрос к нам: что наш tier 1 и где мы читаем код руками? (У спикера: бизнес-метрики, бюджеты, денежные потоки.)
СТУПЕНЬ 3 / ИЕРАРХИЯ
Иерархия влияния на агента

Код > Skills > CLAUDE.md

Самый сильный сигнал — сам код: что в нём написано, то агент и повторит. Skills — операционные процедуры поверх. Длинные правила в CLAUDE.md два верхних этажа переопределяют почти полностью.

Следствие 1

Чистый код сегодня предотвращает плохую генерацию завтра. Качественный код — это обучающий материал для каждой следующей фичи.

Следствие 2

Прототипы и черновики — вне боевого репозитория. Паттерны — в скилы (например, retry во внешние вызовы): один раз описал — везде одинаково. Design doc перечитывать — агент про него забывает.

СТУПЕНЬ 3 / СКИЛЛЫ · данные из доклада
Скиллы всерьёз

Не пишутся руками — генерируются и меряются

Готовые — по областям

Frontend — anthropics/skills · Backend — claude-skills (66 скилов) · Security — Trail of Bits, Sentry · Testing — TDD + debugging (A. Osmani). Каталоги: awesome-agent-skills — 1000+.

Генерируй, не пиши

«Большой скил руками уже не напишешь». Pipeline: draft генерит агент → посмотрел → итерации через evals. Тот же Raptor 1→2→3, только для скилов. Скил — за ~10 минут; эксперт — на постановке и валидации.

Evals — минимальная гигиена

Скил может отлично выглядеть на бумаге и ничего не менять в поведении. Реальный сигнал — только A/B: задача со скилом и без, много прогонов (это делает Tessl). Скилов 30 — без evals не отличишь полезные от мёртвого груза.

Мета-пример из доклада: спикер собрал саму презентацию доклада AI-скилом, включая картинки, — ~20 минут вместо трёх часов. Наш мост: Jira/Jenkins/спека-скилы у нас уже есть — следующий уровень зрелости это evals.
СТУПЕНЬ 3 / ТЕСТЫ · данные из доклада
Тесты и умные моки

Если агент не может запустить тесты — он их не запустит

Скажет «оставлю тебе запустить» — и качество реализации падает. Петля проверки должна крутиться без человека.

«Большую часть времени мы теперь — manual QA, который автоматизирует сам себя».

Полная пирамида

Код : тесты ≈ 1 : 2. Unit — тысячи, быстрые → integration/API/UI → e2e всё замокано → smoke против реального прода.

Умные моки

База и внешние системы замоканы со всеми edge-кейсами. Цель: спека крутится с автопрогоном всех тестов за ~20 минут.

Наш стек: на Java петля и так дорогая (Maven, интеграционные) — моки и быстрые прогоны критичнее вдвойне; связка с IDEA MCP из основного доклада.

СТУПЕНЬ 3 / ЦЕНА · данные из доклада
Честно: цена подхода

Это не «стало легче». Это другая нагрузка

Когнитивная нагрузка выросла

«Сильно возросла. Намного». День на 30K строк — «после него ты овощ»: пять часов думания подряд вместо двух недель с кофе. Помогает спорт и хобби — серьёзно.

Проектирование разрослось

«Пачка A3-листов с диаграммами — сложность уже не держится в голове». Скилы-иллюстраторы обновляют архитектурные диаграммы автоматически.

Сильные инженеры — нужнее

Это не «AI заменит разработчиков»: планка поднялась, потребление растёт (парадокс Джевонса). Потребность в сильных инженерах «намного-намного больше».
