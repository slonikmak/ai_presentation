Тулы

Руки агента

Скиллы

Упакованный рабочий процесс: «как загружать issue», «как делать snapshot replay». Это markdown с инструкцией, не магия. Агент подключает их по ситуации.

Рефлексия

Обратная связь

Агент отмечает, чего не хватило в процессе. Эти заметки — сырьё для новых скиллов и инструкций. Цикл замыкается, если их не терять.

В этой сессии агент использовал пять скиллов — и оставил десять заметок-рефлексий. Обе нити пройдут через весь доклад.

Задача из Jira

Выглядит как CSS-задачка

ES-3772: дать авторам PDF-шаблонов управлять отступом между номером главы и заголовком в ToC, ToF и ToT через styles_toc/tof/tot.css.

Человек → агент
«Загрузи issue ES-3772, изучи и предложи вариант реализации. Эта фича связана с .tasks/es-2676-…»

Спойлер: «добавить CSS-свойство» обернётся изменением DOM-контракта генерации и тремя регрессиями совместимости.

ДНЕВНИК / 01 · ВХОД
Шаг первый — не код

Сначала собрать контекст

скиллjira-issue

Пользователь дал ссылку на Jira — агент не пересказывает URL по памяти, а активирует локальный процесс загрузки issue. Формулировка задачи попадает в работу дословно.

тул.tasks архив

Параллельно читает task-доки родственной задачи ES-2676. Оттуда — готовый архитектурный вывод: номер и заголовок нельзя держать одной строкой, их нужно разделять в DOM.

Цитата из дневника: «Без старой задачи можно было бы начать менять не тот слой пайплайна». Контекст команды — Jira плюс локальные доки — отработал до первой строчки кода.
ДНЕВНИК / 02 · ИССЛЕДОВАНИЕ
Исследование пайплайна

Точка изменения — не там, где похожая старая задача

ES-2676 чинила заголовки в теле документа. Но ToC/ToF/ToT строятся позже — из закладок WeasyPrint, где номер и заголовок уже склеены в один плоский linkText.

Шаблонам физически нечего стилизовать: в DOM один текстовый узел.

html_to_pdf_convert_with_weasyprint.py
def generateOutlineStr(...):
  # page.bookmarks → один плоский linkText
  ...

# toc_template.html / tof / tot:
%(indent)s%(linkText)s

# урок: смотреть реальный runtime path,
# а не «логически похожий» код
ДНЕВНИК / 03 · ПРОЕКТИРОВАНИЕ
скилл · implementation-spec

Спека держит границы

Главная идея: шаблоны получают не один linkText, а структурные плейсхолдеры numberHtml + titleHtml.

  • /Контракты явно: runtime, template, compatibility, validation — четыре секции, на которые сессия потом ссылается.
  • /Non-goals спасают scope: не трогать API, storage, HTML/DOCX export, Pandoc-нумерацию.
  • /Documentation-first — не бюрократия: цитата из дневника — «это помогло удержать границы задачи».
ДНЕВНИК / 03 · ПРОЕКТИРОВАНИЕ
скилл · implementation-plan

Три стадии, тесты — первыми

01

Тесты фиксируют контракт

Regression-тесты на split до любого production-кода. Должны упасть по ожидаемой причине.

02

Runtime split

Helpers в processing-скрипте: разделение bookmark label на номер/префикс и заголовок.

03

Шаблоны и CSS

Новые плейсхолдеры в ToC/ToF/ToT templates, аддитивные селекторы в default CSS.

Коммиты группируются в том же порядке. Риски и митигации — заранее, в самом плане.

ДНЕВНИК / 04 · РЕАЛИЗАЦИЯ
Красный → зелёный

Сначала тест, потом helper

  • /1.2 Introduction and Scope → 1.2 + заголовок

ДНЕВНИК / 05 · ПРОВЕРКА
скилл · export-debug-wsl

Проверить не «как выглядит», а что попало в рантайм

  • /Processing-скрипты синхронизируются в WSL /conversion_utils.
  • /Экспорт снимается в snapshot: workspace со всеми шаблонами, CSS и входным документом.
  • /Snapshot можно переиграть локально стадия за стадией — вплоть до фактического HTML.
Из дневника: «export-debug snapshot — очень сильная техника. Он позволяет проверить не только „PDF выглядит хорошо", но и что именно попало в runtime workspace».
ДНЕВНИК / 05 · ПРОВЕРКА
Минимальная фикстура

Один документ — все три поверхности

  • /Два top-level заголовка и подзаголовок — для ToC
  • /Таблица с caption — для ToT
  • /PlantUML-диаграмма с caption — для ToF
  • /Заголовки нарочно длинные — spacing и переносы видно глазами

Документ остался артефактом в task folder — QA потом использует его же. Проверочный материал — тоже результат работы, не одноразовый мусор.

ДНЕВНИК / 06 · РЕГРЕССИЯ 1
Первый экспорт прошёл. И тут —

Человек видит product-риск, которого агент не видел

Человек → агент
«Сейчас в toc если не задать стили, то номер сливается с текстом. Это ломает старые шаблоны пользователей».
Регрессия №1

Старый CSS не знает новых селекторов

Новая DOM-структура сама по себе не даёт отступа. У клиентов шаблоны написаны до этой фичи — для них номер прилипает к заголовку.

Фикс

Fallback-разделитель в рантайме

Если split произошёл — title-фрагмент начинается с разделителя. Старые шаблоны получают читаемый отступ без единой правки.

ДНЕВНИК / 06 · РЕГРЕССИЯ 1
Урок №1

Unit-тест подтверждает строку. Рендерер живёт своей жизнью

Fallback с обычным пробелом прошёл тесты — а реальный PDF пробел схлопнул. Замена на &nbsp;, обновление тестов.

Из дневника: «Unit test может подтвердить строку, но browser/PDF renderer может вести себя иначе. Для layout-регрессий нужен runtime/snapshot feedback».
fallback evolution
# попытка 1 — тесты зелёные, PDF плохой
<span class="toc-section-title"> Introduction</span>

# попытка 2 — тесты зелёные, PDF хороший
<span class="toc-section-title">&nbsp;Introduction</span>
ДНЕВНИК / 07 · РЕГРЕССИОННЫЙ ШАБЛОН
Агент понял не так — человек уточняет

«Я имел ввиду проверить регрессию»

былоАгент усилил positive case

Понял «проверь второй кейс» как «сделай отступ больше для ToF» — добавил margin-right: 1.2em в стилизованный шаблон.

сталоСимуляция старого шаблона

Отдельный template-regression-no-tof-properties: из CSS удалены новые селекторы. Единственный механизм разделения — runtime fallback. Это и есть мир клиента, который шаблон не обновлял.

Дешёвое уточнение в диалоге сэкономило ложную проверку. Постановка проверки — совместная работа: агент готов проверять, но что проверять — пришло от человека.

ДНЕВНИК / 08 · РЕГРЕССИЯ 2
скиллы · systematic-debugging + export-debug-wsl

Дойти до фактического HTML

Replay снапшота в WSL показал: split вообще не сработал для h1. WeasyPrint отдаёт bookmark без пробела — 1Minimal…, а парсер ждал пробел. Для h2 (1.1 …) всё работало — поэтому ошибка пряталась.

Тест на форму 2Introduction → парсер поддерживает обе формы → sync в WSL → экспорт: «готово. выглядит хорошо».

Из дневника: «Агент не должен „доказывать", что код правильный, если PDF показывает обратное. Нужно дойти до фактического HTML, который попал в renderer».
wsl · replay snapshot
$ python html_to_pdf_convert…py --replay

# фактический generated ToC HTML:
<span class='toc-section-title'>
  1Minimal Export-Debug Document…
</span>

# bookmark от WeasyPrint для h1:
"1Minimal Export-Debug Document" ← нет пробела
ДНЕВНИК / 09 · QA
Техническая работа → проверяемый процесс

Отчёт для QA

Из task-доков агент собирает testing-report: что изменилось, какие варианты шаблонов проверять, что считать правильным выводом, а что — регрессией.

  • /Positive: стилизованный шаблон заметно увеличивает отступ
  • /Regression-safe: старый шаблон без новых свойств остаётся читаемым
  • /Провал: любой слитый вывод вида 1Introduction

Плюс короткий Jira-комментарий без локальных путей — для тех, кто видит только тикет.

ДНЕВНИК / 10 · COMMIT
Git hygiene

Грязное дерево — норма. Коммит — только свой scope

В worktree лежали чужие изменения — DictionaryStorageTest.kt, agents-workflow/. Агент их не тронул: ни в коммит, ни «прибрать».

Из дневника: «Dirty worktree — нормальное состояние реальной разработки. Агент должен уметь коммитить только свой scope, не „убирая" чужие изменения».
git
$ git status
modified: DictionaryStorageTest.kt  ← чужое, не трогаем
untracked: agents-workflow/        ← чужое, не трогаем

$ git commit -m "[ES-3772] Fix PDF ToX spacing"
6b8b91806 → origin/development
ДНЕВНИК / 11 · РЕГРЕССИЯ 3
Push — не конец

QA-шаблон обошёл новый контракт целиком

Человек → агент · после пуша
«Я загрузил кастомный шаблон с qa и сделал экспорт. Результат: текст toc снова слит — 1Minimal Export-Debug Document For ES-3772».
Находка

Старый placeholder

В кастомном toc_template.html — легаси %(linkText)s. Никаких numberHtml/titleHtml: fallback внутри новых плейсхолдеров не мог помочь.

Фикс

Legacy через тот же split

Сначала тест, который точно воспроизводит QA issue и падает. Потом linkText тоже строится через split — старые шаблоны получают 1&nbsp;Title.

Урок

Совместимость — многослойная

CSS-свойства, поведение рендерера, template placeholders — три разных слоя легаси. Ни один не виден из unit-тестов нового кода.

Второй scoped commit: [ES-3772] Preserve legacy ToC spacing → origin/development. Ran 9 tests… OK.

Второй агент читает артефакты снаружи

Рефлексия над рефлексией

  • /Общий корень регрессий: проверка шла на синтетике. Прикреплять к задаче реальные шаблоны, на которых шла проверка.
  • /Фикстуры кодируют допущения агента: тесты строить от runtime-данных, не от воображаемых примеров.
  • /Фикс нарушил spec — spec не догнал: обновлять спеку в том же цикле, что и код.
  • /Рефлексии не должны умирать в скобках: по каждой — скилл / инструкция / отбросить.
  • /Успех документировать как провал: эталонные снимки в task folder, а не «вроде ок».

Изнутри сессии видны тактические улучшения; системное — видно со стороны. Этот шаг дешёвый: второй агент, готовые артефакты. Справа — дневник, раздел 13.

Зачем всё это

Каждая сессия делает следующую дешевле

1 · Сессия

Скиллы работают

Агент переиспользует записанные процессы команды и оставляет артефакты.

2 · Рефлексия

Своя + внешняя

Заметки изнутри сессии и взгляд второго агента по артефактам.

3 · Решение

Скилл / инструкция / отбросить

Каждый блок рефлексии разбирается явно — ничего не умирает в квадратных скобках.

4 · Следующая сессия

Стартует выше

Новые скиллы и инструкции уже в репозитории — связка «человек + агент» дорожает медленнее, чем умнеет.

Это про улучшение процесса человек + агент, а не про отчётность. Хорошо, когда улучшения предлагает сам агент — по итогам рефлексии.
